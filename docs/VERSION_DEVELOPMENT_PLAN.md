# A 股短线选股应用版本开发计划

## 1. 开发规划

### 1.1 核心需求

- 根据当天收盘后的数据，依照实盘选手的买卖逻辑，判断明日他们大概率会买入的股票
- 输出选股名单，不包含可视化功能
- 数据来源希望是同花顺

### 1.2 技术方案

#### 1.2.1 同花顺数据获取

- **核心技术**：Python + Requests + BeautifulSoup/Scrapy
- **数据类型**：
  - 股票基本信息（代码、名称、行业、概念）
  - 当日收盘数据（收盘价、成交量、换手率、涨跌幅）
  - 历史 K 线数据（日线）
  - 板块数据（板块涨幅、涨停股数量）
  - 资金流向数据（主力资金、散户资金）

#### 1.2.2 法律风险分析

- 遵守同花顺的用户协议
- 非商业用途
- 合理使用，限制请求频率
- 数据来源声明

## 3. Git 工作流

### 3.1 分支管理

- **主分支**：`main` - 稳定版本
- **开发分支**：`develop` - 开发中版本
- **功能分支**：`feature/xxx` - 单个功能开发
- **修复分支**：`fix/xxx` - bug 修复

### 3.2 提交规范

使用 conventional commits，格式如下：

```
<type>(<scope>): <description>

<body>

<footer>
```

**类型（type）**：

- `feat`：新功能
- `fix`：bug 修复
- `docs`：文档更新
- `style`：代码风格调整
- `refactor`：代码重构
- `perf`：性能优化
- `test`：测试添加或修改
- `chore`：构建过程或辅助工具的变动

### 3.3 版本发布流程

1. 完成功能开发和测试
2. 合并到`develop`分支
3. 进行集成测试
4. 合并到`main`分支
5. 打 tag，如`v1.0.0`
6. 编写发布说明

## 4. 测试流程

### 4.1 单元测试

- 每个功能模块开发完成后进行单元测试
- 使用 Python unittest 或 pytest 框架
- 测试覆盖率 ≥ 80%

### 4.2 集成测试

- 模块集成后进行集成测试
- 测试模块间的交互
- 确保数据流转正常

### 4.3 系统测试

- 完整系统开发完成后进行系统测试
- 测试系统的稳定性和性能
- 验证端到端功能

### 4.4 回归测试

- 每次修改后进行回归测试
- 确保原有功能不受影响
- 验证 bug 修复的有效性
